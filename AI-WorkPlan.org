* Stage 5: Skills Integration

Goal: Distribute Claude Code skills alongside AI collaboration templates.

Skills are self-contained, filesystem-based packages of task-specific
instructions (each a folder with a =SKILL.md=) that Claude loads on-demand
when the current task calls for them. The skill's description sits in
context by default; the full file is read only when relevant.

Two variants:
- *Anthropic pre-built skills* (e.g. =xlsx=, =docx=, =pptx=, =pdf=) — no
  distribution needed; reference by name.
- *Custom skills* — user/org-defined; these are what we distribute through
  the templates tree.

The natural fit with startAiActivity's existing =mother/= + activity-specific
override pattern:
- =mother/_skills/= — skills available to every activity (baseline).
- =<activity>/_skills/= — activity-specific skills (e.g. a =bisos-pip-release=
  skill under =bisos-pip/_skills/= for version bumps and PyPI publishing).

=initiate= would symlink =.claude/skills/= the same way it currently
symlinks =.claude/settings.json= and =.claude/commands=. =deClaudify=,
=aiSuspend=, and =aiResume= all need to handle the new =.claude/skills=
symlink the same way they handle the other =.claude/= entries.

* Stage 6: Subproject Overlays via initiateSub

Goal: A separate command that installs only the subproject-specific state
(=CLAUDE.md=, =AI-Activity.org=, =AI-DevStatus.org=, =AI-WorkPlan.org=) in
a subdirectory, requiring an already-=initiate=-d ancestor. The invariants
(=AI-AGENTS.org=, =AI-WORKFLOW.org=, =.claude/=) are inherited from the
ancestor via Claude Code's walk-up.

Motivation: nested projects should not re-install the invariants at every
level. Doing so wastes context (Claude reads the same =AI-AGENTS.org= twice)
and creates ambiguity about which activity applies. A slim overlay keeps
subproject state local without duplicating the base.
