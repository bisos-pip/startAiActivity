Key changes since last status:

- *CLAUDE.md is safe-copied, not symlinked (Stage 11):* Claude Code
  resolves symlinks before reading, so =@./AI-Activity.org= inside a
  symlinked =CLAUDE.md= would resolve against the templates directory
  instead of the project. =initiate= and =initiateSub= now safe-copy
  =CLAUDE.md= (no provenance line --- pristine template content).
  =deClaudify= removes it whether symlink (legacy) or file (new).
  =aiSuspend= renames it aside; =aiResume= promotes the =.dormant=
  back or re-copies from templates. Mode detection in =aiResume=
  and =initiateSub= switched from CLAUDE.md-symlink-target to parent
  =AI-WORKFLOW.org= walk-up (still an invariant symlink).
- *New =refresh= command:* Re-copies =CLAUDE.md= from templates.
  Detects base vs sub by walking up for a parent =AI-WORKFLOW.org=
  symlink under templatesBase. Upgrades legacy symlinked installs by
  unlinking then copying. Only touches =CLAUDE.md= for now; per-project
  files and =--noLink= files are never touched.
- *=--noLink= override for symlinked files:* =initiate= and =initiateSub=
  now accept =--noLink=<filename>=. When present, the named file is
  safe-copied instead of symlinked. Primary use case: =--activity=custom=
  installs where the developer wants to own =AI-Activity.org= as a
  per-project file rather than a symlink to the shared template.
- *=custom= activity in all three templates trees:* Added =custom/= to
  BISOS, bxexamples, and T-Mobile fork. Intended install form:

- *Provenance line in safe-copied editable files:* =initiate= and
  =initiateSub= now write a multi-line org-comment provenance block

* What Works

- =startAiActivity.cs -i examples= — shows all available activities when
  templates base is configured; shows placeholder when not yet set.
- =startAiActivity.cs -i userConfig_set --parName=templates --parValue=<path>=
  — persists templates base to =~/.config/bisos/startAiActivity.cs/templates/value=.
- =startAiActivity.cs -i userConfig_get --parName=templates= — reads and
  displays current value.
- =startAiActivity.cs -i initiate --activity=bisos-pip= — installs templates
  into current directory. Now also installs =.claude/skills/= symlink when
  the activity ships a =_skills/= directory (bisos-pip currently ships 5
  skills — see Architecture).
- =startAiActivity.cs -i initiate --root=repo --activity=bisos-pip= — installs
  at git repo root.
- =startAiActivity.cs -i initiateSub --activity=bisos-pip= — installs slim
  subproject overlay in current directory. Requires an ancestor to have been
  =initiate=-d first (walks up looking for a startAiActivity-signature
  =CLAUDE.md=). Does NOT install =AI-WORKFLOW.org=, or
  =.claude/= (all inherited from the ancestor).
- =startAiActivity.cs -i deClaudify= — permanently removes all AI files from
  current directory.
- =startAiActivity.cs -i aiSuspend= — stashes editable files as =.dormant= and
  removes symlinks/=.claude/= from current directory.
- =startAiActivity.cs -i aiResume= — restores =.dormant= files and re-installs
  symlinks/=.claude/=; infers activity automatically.
- Auto-default for templates: if =/bisos/apps/defaults/ai-templates= exists,
  it is written to the user-config store on first run (via =parDictAdd= hook).
- =userConfig_csu= is general-purpose in =bisos.b=; any pycs package can use it
  by adding ='bisos.b.userConfig_csu'= to its =csuList= and tagging params with
  =parPermanence="userConfig"=.
- Skills installation: when the target activity has a =_skills/= directory,
  =initiate= symlinks it to =.claude/skills/= so Claude Code discovers the
  skills automatically. Verified end-to-end against the 5 bisos-pip skills.
- Lifecycle-command skills handling: =aiSuspend= removes =.claude/skills=,
  =aiResume= re-installs it, =deClaudify= removes it — same shape as
  =settings.json= and =commands= handling.
- Subproject overlays: =initiateSub= walks up looking for an initiated
  ancestor and installs a slim overlay (no =AI-WORKFLOW.org=,
  =.claude/=). Precondition uses symlink-target check against
  =templatesBase= — no content parsing. Nested =initiateSub= works
  (parent → child → grandchild verified).
- Mode-aware lifecycle: =aiSuspend= stashes =CLAUDE.md= → =CLAUDE.md.dormant=
  (preserving symlink target as mode signature); =aiResume= reads it to
  choose base vs sub restore path.
- End-to-end verification harness: =py3/test/verify.sh= exercises the full
  command surface (userConfig, initiate, deClaudify, aiSuspend/aiResume with
  mode-detection log, initiateSub positive-and-negative cases) and exits 0
  from a clean start to a clean end. The negative case for =initiateSub= runs
  in a fresh =/tmp= dir with no ancestor CLAUDE.md and expects a visible
  =EH_Problem_UsageError= refusal.

* Key Design Decisions

- *=userConfig_csu= in =bisos.b=, not this package* — the persistent user-config
  CSU is general-purpose and belongs in the PyCS-Core library. Any bisos-pip
  package that needs persistent config can reuse it without copying.

- *=parDictAdd= hook for auto-defaults* — rather than a separate =initUserConfig=
  command, the default is written to the store inside =parDictAdd= itself when
  =parPermanence="userConfig"= and the store is empty. This fires at runtime via
  =g_extraParams=, not at import time. The hook is in =bisos.b/cs/param.py= so
  it is available to all PyCS packages.

- *Precedence-based parDefault probe* — =parDefault= for =templates= is
  set by =_detectTemplatesDefault()=, which probes two well-known paths
  in order: =~/aiActivityTemplates= (canonical per-user clone from the
  templates repo README) first, then =/bisos/apps/defaults/ai-templates=
  (BISOS-provisioned) second. First existing path wins; if neither
  exists, =parDefault= is =None= and the user must =userConfig_set=
  explicitly. User-first ordering means a personal fork cloned to the
  canonical name transparently takes over from the BISOS system copy
  --- no explicit =userConfig_set= needed. This avoids silently storing
  a non-existent path and honors user intent when both are present.

- *=csuListProc= pattern: elisp block is mandatory* — the
  =b:py3:cs:framework/csuListProc= dblock reads =b:py:cs:csuList= from the
  elisp =#+BEGIN_SRC emacs-lisp= block immediately above it. The generated
  =g_extraParams= calls only =cs.csuList_commonParamsSpecify= — not the csxu's
  own =commonParamsSpecify=, which is wired separately by the framework.

- *=README.org= is the only authored source; =py3/README.rst= is generated*
  by =py3/pypiProc.sh -i fullPrep= (invoked directly or via
  =fullPrepBuild forPypi=). Never hand-edit =README.rst= — edits are
  overwritten at release time. Between releases, =README.rst= may be stale
  relative to =README.org=; that's expected and harmless.

- *=mother/= not =bystar/=* — the baseline templates directory is named =mother/=
  in the current templates repo layout. The =excludedDirs= set in =examples=
  uses =mother= accordingly.

- *=deClaudify= vs =aiSuspend= distinction* — =deClaudify= is permanent cleanup
  (deletes editable files); =aiSuspend= is reversible suspension (renames to
  =.dormant=). Both operate on the current directory only — no =--root= param,
  because the files are already present and discoverable.

- *=.dormant= suffix for stashed files* — chosen over a =Stashed-= prefix or
  a separate directory. The suffix keeps files in the same directory (no path
  changes), sorts them near their originals, and is unambiguous to both humans
  and tooling. =aiResume= looks for exactly these suffixed names.

- *Activity inference in =aiResume=* — first tries the =AI-Activity.org= symlink
  target (parent dir name = activity); falls back to the =Activity:= header line
  in the =AI-WorkPlan.org= dblock. This means =aiResume= works correctly even
  after =aiSuspend= has removed the symlink, as long as the =.dormant= WorkPlan
  is present.

- *Skills at activity root, not under =_claude/=* — skills live in
  =<activity>/_skills/=, not =<activity>/_claude/skills/=. Rationale: skills are
  semantically distinct from =.claude/= machinery — they're knowledge, not
  harness config. Placing them at the activity root keeps them independently
  visible to humans and other tools; the installed location under
  =.claude/skills/= is just where Claude Code looks. =initiate= symlinks
  =<activity>/_skills/= to =.claude/skills/= to bridge the two.

- *Subproject overlays use Claude Code walk-up, not local hooks* — an
  =initiateSub= directory has a slim =CLAUDE.md= that imports only its
  local trio. The invariants come from an ancestor's =CLAUDE.md= via
  Claude Code's built-in upward walk. This avoids relying on manual
  prompting or session hooks — discovery is fully automatic once files
  are on disk.

- *Symlink-target as mode signature* — =aiSuspend= renames =CLAUDE.md=
  to =CLAUDE.md.dormant= (preserving the symlink target) rather than
  removing it. =aiResume= reads the target with =readlink= + resolve and
  compares against =<templatesBase>/mother/initiateSub/CLAUDE.md= to
  distinguish base from sub. Chose this over storing mode in a dblock
  or requiring a =--sub= flag: (a) zero user burden, (b) tolerates
  hand-edits (only the symlink target matters), (c) survives filesystem
  copies and git checkouts.

- *=--activity= required on =initiateSub=* — no inheritance-from-parent
  default. Rationale: a subproject can legitimately be a different
  activity than its parent (e.g., a =blee-panels= subproject inside a
  =bisos-pip= repo), and inference would silently pick the wrong one
  in that case. Explicit is better.

- *Nested =initiateSub= allowed* — both base and sub CLAUDE.md symlinks
  land under =templatesBase=, so the precondition check treats them
  equivalently. Grandchild's walk-up hits child's =CLAUDE.md=, which
  points at =mother/initiateSub/CLAUDE.md= — still under =templatesBase=,
  precondition satisfied.

- *Splitting =AI-Activity.org= into orientation + skills* — the pre-Stage-5
  =bisos-pip/AI-Activity.org= was 456 lines of always-loaded context, of which
  ~60% was procedural knowledge only needed for specific tasks (writing CS
  commands, authoring READMEs, releasing to PyPI). Splitting cut the baseline
  to 154 lines and moved ~916 lines of procedural content into 5 skills that
  load on demand. Original preserved as =AI-Activity.org.pre-skills-backup=.
  Trade-off: skills are a Claude Code-only feature; a human reading
  =AI-Activity.org= outside Claude Code will see references but not content.
  Mitigation: the "Available Skills" table in =AI-Activity.org= names each
  skill's file path, so a human can open it directly.

- *This repo functions as a working example of the full AI Activity
  workflow.* The presence of =AI-WorkPlan.org_archive= alongside
  =AI-WorkPlan.org= demonstrates the org-archive pattern in action ---
  completed stages are moved to the archive file (via org-mode's
  =org-archive-subtree=) to keep the active WorkPlan focused on current
  work. Other artifacts in this repo (=verify.sh=, the skills folder,
  =AI-DevStatus.org='s update discipline) are similarly load-bearing
  as reference implementations of what the templates enable.

- *=CLAUDE.md= imports the four subordinate files directly; no
  =AI-AGENTS.org= indirection.* The historical layer added no
  functional value --- =AI-AGENTS.org= was aspirationally
  vendor-neutral but 100% Claude-specific in content, and every user
  hop through it just made the graph deeper. Merging =CLAUDE.md= to
  import =AI-WORKFLOW.org= / =AI-Activity.org= / =AI-DevStatus.org= /
  =AI-WorkPlan.org= directly eliminates one file, one symlink, one
  install step, and one layer of indirection without loss. Legacy
  projects with an =AI-AGENTS.org= symlink still get correctly cleaned
  up by =deClaudify= and =aiSuspend= (both retain the entry in their
  cleanup lists with an explanatory comment).

- *=--noLink= is a CLI flag, not a per-activity config file.* When
  the need for per-project safe-copy of =AI-Activity.org= (and
  possibly other files) emerged with the =custom= activity, two
  designs were considered: (a) a per-activity =_activity-config.org=
  in the templates tree declaring which files should be safe-copied,
  or (b) a =--noLink=<file>= CLI flag on the invocation. Chose (b)
  because it preserves the design principle that *templates should
  stay passive* --- a template tree is a filesystem of examples, not
  a configuration layer requiring engine-side interpretation. (a)
  would have put behavior specification in two places (config file
  plus optional CLI override) with precedence rules to reason about.
  (b) is more general (works on any file, not only pre-declared
  ones), simpler (one mechanism), and honest (the choice is made at
  invocation time by the human who knows what they want). The
  trade-off: the user has to remember the flag. Mitigated by
  advertising =--noLink=AI-Activity.org= in the =examples= menu
  right after each =--activity=custom= line, and by keeping the
  =custom= activity itself rare (used only when no standard activity
  fits).

- *=CLAUDE.md= is safe-copied, not symlinked.* Claude Code resolves
  symlinks before reading a file, so =@./AI-Activity.org= inside a
  symlinked =CLAUDE.md= resolves relative to the templates directory
  (where the symlink target lives), not the project. That silently pulls
  in the wrong =AI-Activity.org= --- discovered in a fresh Claude session
  reading site-status-scripts. Fix: =initiate= and =initiateSub=
  =shutil.copy2= =CLAUDE.md= from templates instead of =symlink_to=; no
  provenance line is written (CLAUDE.md must remain equivalent to a
  symlink in content). The new =refresh= command is the escape hatch:
  re-copies =CLAUDE.md= from templates on demand, upgrading any legacy
  symlinked install in place. Alternatives considered: (a) drop =@./=
  imports from CLAUDE.md and rely on prose --- rejected as fragile;
  (b) absolute =@/full/path= imports --- doesn't work for per-project
  files (=AI-Activity.org=, etc.); (c) two-level wrapper --- doesn't
  solve the underlying resolution. Trade-off: templates changes to
  CLAUDE.md don't propagate automatically to installed projects ---
  mitigated by =refresh=. Mode detection in =aiResume= and =initiateSub=
  switched from CLAUDE.md-symlink-target to parent =AI-WORKFLOW.org=
  walk-up under templatesBase; =AI-WORKFLOW.org= is still symlinked
  (it has no back-imports, so symlink resolution is harmless).

- *Provenance is inert text, not a dblock.* The provenance line
  written by =initiate= / =initiateSub= into =AI-DevStatus.org= and
  =AI-WorkPlan.org= (date, user, templates base, =sys.argv=) is a
  plain multi-line org-comment block, not a dblock of any signature.
  Dblocks have two implementations (Python via =bisos.pyDblock=, and
  elisp via =org-dblock-update= inside Emacs). Elisp dblock
  regeneration cannot recover invocation context --- it runs long
  after =initiate= exited, in a shell it never observed. Any dblock-
  based approach would either only work under the Python expander,
  or would need parallel elisp code reaching into a persistence
  store. Provenance is *birth-certificate* data, correct only at
  install time; inert text written once matches the semantics
  exactly. Placement is after the last =#+= header directive at the
  top of the file, robust to header reordering.
