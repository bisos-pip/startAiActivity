* What Works

- =startAiActivity.cs -i examples= — shows all available activities when
  templates base is configured; shows placeholder when not yet set.
- =startAiActivity.cs -i userConfig_set --parName=templates --parValue=<path>=
  — persists templates base to =~/.config/bisos/startAiActivity.cs/templates/value=.
- =startAiActivity.cs -i userConfig_get --parName=templates= — reads and
  displays current value.
- =startAiActivity.cs -i initiate --activity=bisos-pip= — installs templates
  into current directory. Now also installs =.claude/skills/= symlink when
  the activity ships a =_skills/= directory (bisos-pip currently ships 5
  skills — see Architecture).
- =startAiActivity.cs -i initiate --root=repo --activity=bisos-pip= — installs
  at git repo root.
- =startAiActivity.cs -i deClaudify= — permanently removes all AI files from
  current directory.
- =startAiActivity.cs -i aiSuspend= — stashes editable files as =.dormant= and
  removes symlinks/=.claude/= from current directory.
- =startAiActivity.cs -i aiResume= — restores =.dormant= files and re-installs
  symlinks/=.claude/=; infers activity automatically.
- Auto-default for templates: if =/bisos/apps/defaults/ai-templates= exists,
  it is written to the user-config store on first run (via =parDictAdd= hook).
- =userConfig_csu= is general-purpose in =bisos.b=; any pycs package can use it
  by adding ='bisos.b.userConfig_csu'= to its =csuList= and tagging params with
  =parPermanence="userConfig"=.
- Skills installation: when the target activity has a =_skills/= directory,
  =initiate= symlinks it to =.claude/skills/= so Claude Code discovers the
  skills automatically. Verified end-to-end against the 5 bisos-pip skills.
- Lifecycle-command skills handling: =aiSuspend= removes =.claude/skills=,
  =aiResume= re-installs it, =deClaudify= removes it — same shape as
  =settings.json= and =commands= handling.

* Key Design Decisions

- *=userConfig_csu= in =bisos.b=, not this package* — the persistent user-config
  CSU is general-purpose and belongs in the PyCS-Core library. Any bisos-pip
  package that needs persistent config can reuse it without copying.

- *=parDictAdd= hook for auto-defaults* — rather than a separate =initUserConfig=
  command, the default is written to the store inside =parDictAdd= itself when
  =parPermanence="userConfig"= and the store is empty. This fires at runtime via
  =g_extraParams=, not at import time. The hook is in =bisos.b/cs/param.py= so
  it is available to all PyCS packages.

- *Conditional parDefault* — =parDefault= for =templates= is set to the well-known
  path only if that directory actually exists on the machine. On a machine without
  =/bisos/apps/defaults/ai-templates=, =parDefault= is =None= and the user must
  configure manually. This avoids silently storing a non-existent path.

- *=csuListProc= pattern: elisp block is mandatory* — the
  =b:py3:cs:framework/csuListProc= dblock reads =b:py:cs:csuList= from the
  elisp =#+BEGIN_SRC emacs-lisp= block immediately above it. The generated
  =g_extraParams= calls only =cs.csuList_commonParamsSpecify= — not the csxu's
  own =commonParamsSpecify=, which is wired separately by the framework.

- *=README.org= is the authoritative source* — =py3/README.rst= is derived from
  it. The RST is written to match pandoc/ox-rst output conventions. When both
  need updating, edit =README.org= first, then regenerate or manually sync RST.

- *=mother/= not =bystar/=* — the baseline templates directory is named =mother/=
  in the current templates repo layout. The =excludedDirs= set in =examples=
  uses =mother= accordingly.

- *=deClaudify= vs =aiSuspend= distinction* — =deClaudify= is permanent cleanup
  (deletes editable files); =aiSuspend= is reversible suspension (renames to
  =.dormant=). Both operate on the current directory only — no =--root= param,
  because the files are already present and discoverable.

- *=.dormant= suffix for stashed files* — chosen over a =Stashed-= prefix or
  a separate directory. The suffix keeps files in the same directory (no path
  changes), sorts them near their originals, and is unambiguous to both humans
  and tooling. =aiResume= looks for exactly these suffixed names.

- *Activity inference in =aiResume=* — first tries the =AI-Activity.org= symlink
  target (parent dir name = activity); falls back to the =Activity:= header line
  in the =AI-WorkPlan.org= dblock. This means =aiResume= works correctly even
  after =aiSuspend= has removed the symlink, as long as the =.dormant= WorkPlan
  is present.

- *Skills at activity root, not under =_claude/=* — skills live in
  =<activity>/_skills/=, not =<activity>/_claude/skills/=. Rationale: skills are
  semantically distinct from =.claude/= machinery — they're knowledge, not
  harness config. Placing them at the activity root keeps them independently
  visible to humans and other tools; the installed location under
  =.claude/skills/= is just where Claude Code looks. =initiate= symlinks
  =<activity>/_skills/= to =.claude/skills/= to bridge the two.

- *Splitting =AI-Activity.org= into orientation + skills* — the pre-Stage-5
  =bisos-pip/AI-Activity.org= was 456 lines of always-loaded context, of which
  ~60% was procedural knowledge only needed for specific tasks (writing CS
  commands, authoring READMEs, releasing to PyPI). Splitting cut the baseline
  to 154 lines and moved ~916 lines of procedural content into 5 skills that
  load on demand. Original preserved as =AI-Activity.org.pre-skills-backup=.
  Trade-off: skills are a Claude Code-only feature; a human reading
  =AI-Activity.org= outside Claude Code will see references but not content.
  Mitigation: the "Available Skills" table in =AI-Activity.org= names each
  skill's file path, so a human can open it directly.
